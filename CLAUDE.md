# MLLM Vault 工作规范

## 语言与角色

- 默认使用简体中文进行回复、整理和笔记编写。
- 当前工作区是一个 Obsidian vault，用于 MLLM、多模态学习、世界模型相关的学习与研究整理。
- 目标是在不破坏现有结构的前提下，维护一个清晰、可扩展、适合长期积累的知识库。

## 当前仓库结构

根目录下当前主要目录：

- `raw/`：原始资料、论文、视频相关输入
- `insights/`：提炼后的理解、总结、研究灵感
- `user/`：个人信息与身份相关笔记
- `未命名/`：临时未分类笔记
- `Clippings/`：剪藏内容
- `skills/`：给 Obsidian 可见的 skills 阅读副本
- `commands/`：给 Obsidian 可见的 commands 阅读副本

隐藏但实际生效的 Claude 配置目录：

- `.claude/skills/`
- `.claude/commands/`

## 目录使用约定

### raw/

- 视作输入层。
- 优先放：论文、网页摘录、视频笔记、转录文本、原始材料。
- 除非用户明确要求，不删除用户已有原始资料。
- 可以读取、总结、建立关联，但尽量不要改写其原始内容。

### insights/

- 视作知识提炼层。
- 优先放：
  - 论文摘要
  - 方法总结
  - 研究问题拆解
  - 灵感记录
  - 跨来源综合分析
- 鼓励使用清晰标题和项目符号。
- 较长笔记建议包含摘要。

### user/

- 放与用户身份、背景、长期研究方向直接相关的内容。
- 未明确要求时，不移动、不删除。

### 未命名/

- 仅作为临时区。
- 如果后续整理内容，优先通过建立链接和新建整理笔记来逐步吸收，而不是一次性大规模搬迁。

## 笔记风格

- 标题简洁明确。
- 多用项目符号，便于扫描。
- 适合时使用 Obsidian 内链，例如 `[[某篇论文]]`。
- 对于学习类笔记，优先考虑以下结构：
  - Topic / 主题
  - Key ideas / 核心观点
  - Examples / 示例
  - Questions / 问题
  - Related notes / 相关笔记

## 与 skills / commands 的关系

本仓库已经配置以下项目命令：

- `/ingest`
- `/query`
- `/lint`

它们的真实定义位置是：

- `.claude/skills/`
- `.claude/commands/`

可见副本位于：

- `skills/`
- `commands/`

### skills 书写规范

- 在 `mllm/skills/` 下编写或维护的 skill 文档，注释、说明性文字、使用指引应尽量提供中英双语版本。
- 优先保证中文表达清楚，再补充对应英文，便于本地阅读、分享和后续迁移复用。
- 这里的“双语”主要指面向人阅读的说明内容；代码、路径、命令名、frontmatter 字段名保持原样，不需要为标识符强行翻译。

### /ingest

作用：
- 将原始资料整理为结构化笔记
- 优先从 `raw/` 中读取输入
- 输出优先写入 `insights/` 或用户明确指定的位置

建议行为：
- 为单个来源生成摘要笔记
- 提取概念、方法、问题与启发
- 建立必要的 Obsidian 双链
- 如发现内容冲突，先询问用户

### /query

作用：
- 从当前 vault 中检索相关笔记并回答问题

建议行为：
- 先查看已有笔记，再做综合回答
- 优先引用本地内容
- 如果本地没有相关内容，明确说明，再补充通用知识

### /lint

作用：
- 检查当前知识库结构健康度

建议行为：
- 扫描死链、孤儿页面、未整理内容
- 优先报告问题，再等待用户确认后修复

## 工作原则

- 每次开始时，优先参考 `AGENTS.md`、本文件，以及 `user/user.md`。
- 如果 `user/user.md` 存在，默认先读取它，再开始后续整理、回答或结构调整工作。
- 优先做 Markdown 层面的简单改动。
- 未明确要求时，不进行大规模目录重构。
- 未明确要求时，不删除用户笔记。
- 整理内容时，优先通过新建总结页和建立链接来增强结构。
- 如果引入外部模板仓库的方法，应尽量以“渐进吸收”的方式融入当前 vault。

## 推荐输出位置

当没有更明确指示时：

- 原始输入、论文、摘录 → `raw/`
- 提炼总结、研究思考 → `insights/`
- 个人长期信息 → `user/`
